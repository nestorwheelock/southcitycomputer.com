1. Resumen Ejecutivo

Este documento técnico documenta nuestro viaje experimental para construir el sitio web más rápido posible para un pequeño negocio. A través de optimización sistemática, logramos:

  • Cargas de página 1,500 veces más rápidas (de 2.4 segundos a 1.6 milisegundos)
  • Reducción del 78% en el peso de imágenes (de 20MB a 4.3MB)
  • Tiempos de respuesta sub-milisegundo (36-382 microsegundos) para activos principales
  • 26,000-58,000 solicitudes/segundo de rendimiento en hardware modesto

2. El Caso de Negocio para la Velocidad

2.1 Las Expectativas de los Usuarios Han Cambiado

El 47% de los usuarios ahora esperan tiempos de carga menores a 2 segundos, más estricto que los 4 segundos de hace unos años. El 53% de los usuarios móviles abandonan sitios que tardan más de 3 segundos.

2.2 El Precipicio de la Tasa de Rebote

Tiempo de Carga Tasa de Rebote Cambio
1 segundo 7% Base
3 segundos 11% +57%
5 segundos 38% +443%

2.3 Impacto en la Tasa de Conversión

Los sitios que cargan en 1 segundo logran tasas de conversión del 39%. A los 5.7 segundos, esto cae al 0.6%—una diferencia de 65 veces. Las tasas de conversión de sitios web caen un promedio de 4.42% por cada segundo adicional de tiempo de carga.

3. Metodología

3.1 Sujeto de Prueba

Sitio web de South City Computer presentando:

  • 15 fotografías de alta calidad
  • Contenido bilingüe (Inglés/Español)

3.2 Rondas de Prueba

Ronda 1: Línea Base
├── nginx + Rust API + JPEG desde disco
└── Resultado: 2.4 segundos de carga

Ronda 2: Optimización de Imágenes
├── Convertir todas las imágenes a WebP
└── Resultado: 78% reducción de tamaño

Ronda 3: Optimización Completa
├── Binario monolítico en Rust
├── Todos los activos embebidos en memoria
├── CSS/JS minificados
└── Resultado: 52ms de carga

Ronda 4: Benchmarking de Alto Rendimiento
├── Herramientas de benchmark personalizadas (scc-benchmark, scc-perf-client)
├── Metodología de pruebas de alta concurrencia
├── Mediciones de rendimiento reproducibles
└── Resultado: 1.6ms carga completa, 58,085 sol/s

6. Comparación: Personalizado vs WordPress + CDN

6.1 Realidad del Rendimiento de WordPress

WordPress alimenta el 43% de los sitios web, pero el rendimiento varía significativamente:

  • Carga promedio de WordPress en escritorio: 2.5-3 segundos
  • Carga promedio de WordPress en móvil: 13.25 segundos
  • WordPress ocupa el puesto 15 de 20 CMSs en velocidad

6.2 ¿Puede Cloudflare Arreglar WordPress?

Cloudflare proporciona mejoras significativas para contenido estático, pero no puede arreglar:

  • Renderizado lento del lado del servidor (PHP/MySQL)
  • Sobrecarga de plugins (el sitio promedio de WordPress tiene más de 20 plugins)
  • Latencia de consultas a base de datos
  • Cargas de página grandes

6.3 Cuándo Usar Qué

Elige WordPress cuando:

  • El contenido cambia frecuentemente (se necesitan características de CMS)
  • Usuarios no técnicos gestionan el contenido
  • El tiempo de lanzamiento es crítico

Elige código personalizado cuando:

  • El rendimiento es una ventaja competitiva
  • El contenido es relativamente estático
  • Los rankings SEO son críticos en un nicho competitivo
  • Los costos de memoria/cómputo importan

7. Conclusiones

7.1 Hallazgos Clave

  1. La optimización de imágenes tiene el mayor ROI — La conversión a WebP sola proporciona una reducción del 78% con mínimo esfuerzo.
  2. Rust sobresale para aplicaciones críticas en rendimiento — Más de 58,000 sol/s en hardware básico con tiempos de respuesta sub-milisegundo.
  3. Los CDNs nivelan el campo de juego significativamente — Cloudflare puede hacer que WordPress sea competitivo para contenido estático.
  4. El código personalizado en Rust/Go proporciona rendimiento 100-1000x mejor que lenguajes interpretados con frameworks pesados.

7.2 La Conclusión Final

Para la mayoría de los sitios web, imágenes WebP + CDN de Cloudflare proporciona el 80% del beneficio con el 10% del esfuerzo.

Para nichos competitivos donde cada milisegundo importa, la arquitectura Rust + residente en memoria proporciona un rendimiento sin igual que ninguna cantidad de caché puede replicar.
